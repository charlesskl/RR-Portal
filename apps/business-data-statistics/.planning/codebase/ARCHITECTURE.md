# Architecture

**Analysis Date:** 2026-03-17

## Pattern Overview

**Overall:** Monolithic MVC with SPA frontend (server-rendered shell + client-side routing)

**Key Characteristics:**
- Single Express server serves both API (`/api/*`) and static SPA files (via `public/`)
- Frontend is a Vue 3 SPA loaded from `public/index.html` using CDN-loaded libraries (no build step)
- All frontend code lives in one large file: `public/js/app.js` (1801 lines)
- Backend config-driven pattern: all three departments (`beer`, `print`, `assembly`) share a single `records.js` route handler, parameterized by a `:dept` URL segment
- A central config module (`modules/balance/config.js` + `modules/index.js`) is the single source of truth for field definitions, driving SQL construction, Excel import/export mapping, and frontend rendering

## Layers

**Configuration Layer:**
- Purpose: Define all per-department field schemas (input, expense, calc, structural) and derive runtime data structures from them
- Location: `modules/balance/config.js`, `modules/index.js`
- Contains: Field definitions (label, aliases, type, input/expense/calc flags), DEPT_CONFIG builder, COLUMN_MAP generator, startup validator
- Depends on: Nothing
- Used by: All route handlers, `modules/balance/calc.js`, `routes/import-export.js`

**Database Layer:**
- Purpose: Raw PostgreSQL access with a lightweight ? → $N placeholder conversion
- Location: `db/postgres.js`
- Contains: `query()`, `getOne()`, `getAll()` wrappers around `pg.Pool`
- Depends on: `pg` (node-postgres)
- Used by: All route files, middleware

**Middleware Layer:**
- Purpose: Cross-cutting request concerns applied to routes
- Location: `middleware/auth.js`, `middleware/audit.js`
- Contains:
  - `authenticate` — JWT Bearer token verification, attaches `req.user`
  - `requireStats` — role gate (stats only)
  - `modulePermission(name)` — per-module permission check for entry-role users (reads `user_modules` table)
  - `checkDataLock` — blocks writes if that month/dept is locked (skipped for stats role)
  - `logAction` — writes CRUD events to `audit_logs` table
- Depends on: `db/postgres.js`, `jsonwebtoken`
- Used by: Route handlers

**Route Layer:**
- Purpose: HTTP request handling, business logic orchestration
- Location: `routes/` directory
- Contains:
  - `records.js` — generic dept CRUD + batch delete + dept-level summary (parameterized by `:dept`)
  - `summary.js` — cross-department overview aggregation
  - `import-export.js` — Excel upload/parse/insert and export/generate
  - `auth.js` — login + `/me` endpoint
  - `users.js`, `workshops.js`, `settings.js`, `audit-logs.js`, `backup.js` — admin management
- Depends on: `db/postgres.js`, `middleware/`, `modules/`, `utils/async-handler.js`
- Used by: `server.js` via `app.use()`

**Calculation Layer:**
- Purpose: Stateless pure functions computing derived fields from input data
- Location: `modules/balance/calc.js`
- Contains: `calculateRecord(dept, record)` — takes a raw record object, returns it with all calc fields populated
- Depends on: `modules/index.js` (for expense field lists)
- Used by: `routes/records.js` (POST/PUT), `routes/import-export.js` (during import)

**Frontend Layer:**
- Purpose: SPA UI — all pages, components, routing, and state
- Location: `public/js/app.js` (components), `public/js/api.js` (HTTP client), `public/css/theme.css` (styles)
- Contains: Vue 3 component objects (`LoginPage`, `DeptRecordsPage`, `SummaryPage`, `UserManagementPage`, `SettingsPage`, `WorkshopSettings`, `DataLocks`, `AuditLogs`, `BackupPage`), client-side hash router, field config constants (`DEPT_CONFIG`, `SHARED_*` arrays)
- Depends on: Vue 3, Element Plus, Axios (all CDN), `api.js`
- Used by: Browser only

## Data Flow

**Write Record (POST):**

1. Frontend (`DeptRecordsPage` in `app.js`) calls `API.post('/beer/records', formData)` via `api.js`
2. `server.js` routes to `routes/records.js` POST handler
3. `authenticate` middleware verifies JWT, `checkDataLock` verifies month is unlocked
4. Route calls `calculateRecord(dept, rawBody)` from `modules/balance/calc.js`
5. Computed record is inserted to `beer_records` (or `print_records`/`assembly_records`) via `db/postgres.js`
6. `logAction()` writes audit entry; response returns inserted row

**Read Records (GET):**

1. Frontend calls `API.get('/:dept/records', { start_date, end_date, workshop_id })`
2. Route queries `{dept}_records JOIN workshops` with optional date/workshop filters
3. Returns raw DB rows (no recalculation — calc fields are stored at insert/update time)

**Excel Import:**

1. Frontend sends `multipart/form-data` file upload to `/api/:dept/import`
2. `routes/import-export.js` parses with SheetJS, normalizes column names via `cleanColumnName()`
3. Maps Excel column headers to DB field names using `COLUMN_MAP` (generated by `getColumnMap('balance')` from `modules/index.js`)
4. For each row: calls `calculateRecord()`, inserts to DB, logs action

**State Management (Frontend):**
- No Vuex/Pinia — each page component manages its own `data()` state
- Auth state (`user`, `token`) is shared via `this.$root.user` and `localStorage`
- Route state is managed by `window.location.hash` watched in the root app component

## Key Abstractions

**DEPT_CONFIG (backend):**
- Purpose: Runtime lookup table keyed by dept slug (`beer`/`print`/`assembly`) providing `tableName`, `uniqueInputFields[]`, `uniqueCalcFields[]`, `uniqueExpenseFields[]`
- Source: Derived from `modules/balance/config.js` via `modules/index.js`
- Examples: `modules/index.js` lines 29–44
- Pattern: Built once at startup; routes use `DEPT_CONFIG[req.params.dept]` to get table name and field lists

**DEPT_CONFIG (frontend):**
- Purpose: Parallel field schema for rendering table columns — entirely separate definition from backend's
- Location: `public/js/app.js` lines 6–85
- Pattern: Each dept has `uniqueFields[]` with `shortLabel` (table header), `editable`, `type`, `calculated`, `formula` — these drive column rendering in `DeptRecordsPage`

**COLUMN_MAP:**
- Purpose: Maps Chinese Excel column headers (and aliases) to English DB field names for import
- Location: Generated at `routes/import-export.js` startup via `getColumnMap('balance')`
- Pattern: `{ '总产值/天': 'daily_output', '产值': 'daily_output', '结余金额': '_skip_calc', ... }`

**asyncHandler:**
- Purpose: Express route wrapper that catches rejected promises and returns `{ success: false, message }` 500 responses
- Location: `utils/async-handler.js`
- Pattern: All routes use `asyncHandler(async (req, res) => { ... })` except import-export (uses manual try/catch for richer error handling)

## Entry Points

**HTTP Server:**
- Location: `server.js`
- Triggers: PM2 `npm start` or `node server.js`
- Responsibilities: Register all routes, serve static files, provide `/api/health`, SPA fallback to `public/index.html`

**SPA Bootstrap:**
- Location: `public/index.html` → loads `public/js/api.js`, `public/js/app.js` (CDN Vue/ElementPlus/Axios)
- Triggers: Browser navigation to any `http://localhost:6001/*` URL
- Responsibilities: Create Vue app, register all page components, mount to `#app`, initialize hash-based router

**Config Validation:**
- Location: `server.js` line 6–7: `require('./modules').validateConfig()`
- Triggers: Server startup
- Responsibilities: Verifies all expense fields are also input fields (prevents silent calculation errors)

## Error Handling

**Strategy:** Uniform JSON error envelope `{ success: false, message: string }`

**Patterns:**
- All routes wrapped in `asyncHandler` from `utils/async-handler.js` — uncaught async errors become 500 responses
- Auth errors return 401 with `{ success: false, message: '未登录' }` or `'登录已过期'`
- Permission errors return 403 with `{ success: false, message: '...' }`
- Validation errors (invalid dept, missing body fields) return 400
- Frontend `api.js` intercepts 401 responses globally: clears token + redirects to `#/login`
- Import route uses manual `try/catch` (not `asyncHandler`) to collect per-row errors and return partial success

## Cross-Cutting Concerns

**Logging:** `middleware/audit.js` `logAction()` writes to `audit_logs` table on every create/update/delete/batch_delete. Viewed via `routes/audit-logs.js`.

**Validation:** Dept slug validation in `records.js` via `validateDept` middleware. Config integrity validated at startup by `validateConfig()`. No runtime schema validation library — fields are trusted from config arrays.

**Authentication:** JWT Bearer tokens, 24h expiry, secret from `process.env.JWT_SECRET` (defaults to hardcoded string in development). Roles: `stats` (full access), `entry` (module-gated + data lock-gated).
