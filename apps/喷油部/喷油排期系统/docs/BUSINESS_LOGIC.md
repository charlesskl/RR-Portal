## 第 0 章：系统定位与生态位

### 0.1 印喷部在生产链中的位置

```
[啤机部]  →  [印喷部（本系统）]  →  [装配部]
  注塑/吹塑     印刷/喷涂加工          组装成品
  (上游)         (中游)                (下游)
```

**当前决策（2026-05-20）：**
- ✅ 啤机部、装配部目前**没有系统**，本系统是第一个
- ✅ 本系统在**数据结构上预留接口字段**，但**不实现任何对接逻辑**
- ✅ 未来啤机/装配上线时，按本系统的标准对接（本系统先定标准）
- ✅ **所有订单都是公司内部的委托加工单**——啤机部以"内部客户"身份开单给喷油部，喷油部加工完交给装配部
- ✅ **不存在外部客户直接下单到喷油部的场景**（第 0 章首日决策）

### 0.2 自产订单 vs 外发订单（核心分支）

**关键决策（2026-05-20）：**

| 维度 | 自产订单 | 外发订单 |
|---|---|---|
| 工艺模板 | ✅ 必须套模板，自动展开子件/工序/步骤 | ✅ **不套模板**，订单只是"挂账" |
| 数据结构 | 订单 → 款号 → 子件 → 颜色 → 工序步骤 → 每日实绩 | 订单 → 外发厂 → 收货验收数（单层） |
| 实绩录入 | 每日纸单录入树形结构 | **每日交货数滚动录入**（外发分批返厂，详见 2.5）|
| 排期 | 占用本厂车间产能，进甘特图 | 不占车间，只记录"预计返厂日期" |
| 木桶效应 | ✅ 计算（多子件完工率取 MIN） | ❌ 不适用（只有一个总量） |
| 进日报 | ✅ 进 | ❌ **不进** |
| 成本口径 | 工价 + 油墨成本 | 外发加工费（一口价 HK$/件） |
| 月度对账 | 走部门日报汇总 | 走"外发成本月报"（独立口径） |

### 0.3 上下游接口预留（仅数据结构，不实现）

由于"所有订单都是内部委托加工单"的确认，预留字段比之前简化：

| 字段 | 用途 | 当前 V1 默认值 |
|---|---|---|
| `orders.source_dept` | 入料来源部门 | 默认"啤机部"（未来可能有"其他车间"）|
| `orders.destination_dept` | 出料去向部门 | 默认"装配部"（首日确认） |
| `orders.upstream_order_no` | 上游系统的订单号引用 | V1 = 当前订单号本身（无独立母单号） |
| 啤机收料 API | 啤机部产出 → 印喷部入料 | 不实现 |
| 装配交料 API | 印喷部产出 → 装配部入料 | 不实现 |

**已删除字段**（spec 中曾考虑但被业务推翻）：
- ⚠️ `customer_name` 客户名称 —— 全是内部加工，无外部客户
- ⚠️ `customer_contact` 客户联系人 —— 同上
- ⚠️ `sales_person` 销售员 —— 同上
- ⚠️ `quotation_price` 客户报价 —— 同上
- ⚠️ `parent_order_no` 母订单号 —— 暂不实现跨部门串联，仅用 `upstream_order_no` 留单一引用

### 0.4 关键认知修正记录（防止重蹈覆辙）

读真实订单 `兴信喷油11494.pdf` 之前我们的设计与实际业务的差异，记录在此防止再次走偏：

| 维度 | 原 spec（已推翻） | 真实业务（已确认） |
|---|---|---|
| 业务定位 | 接外部客户订单的生产部门 | 公司内部加工车间，"客户"= 啤机部 |
| 订单来源 | 文员看微信截图录入 | 啤机部开"委托加工合同"，喷油部接收 |
| 订单号 | 系统自动生成 | **由上游系统/上游文员决定**（如 ZWZ2026057），喷油部当字符串接收 |
| 单价 | 工艺师人工核价 | **核价表 = 工艺模板**，单价预设好，自动从模板带出 |
| 数据层级 | 产品 → 部位 → 工序 → 步骤 | **订单 → 款号+版本 → 子件 → 部位 → 颜色 → 工序步骤**（6 层） |
| 部位 | 物理部位（头/身/手/脚） | 「子件」和「部位」**都存在**：子件是套装小件（兔子/青蛙）或单玩具（手），部位是该子件的物理细分（兔子耳朵/身体），单玩具场景下子件 = 部位自指（第 2 天 5-21 修正） |
| 颜色 | 工艺模板的一部分 | 订单维度的生产参数，与工艺模板解耦 |
| 客户管理 | 需要建客户库 | **不需要**（内部加工没有客户概念） |
| 出货 | 发货给客户 | 交给装配部 |
| 订单复杂度 | 每张订单几行 | **每张订单 150+ 行**（17 个子件 × ~10 种颜色） |

**今天我们学到的最重要一条经验：** 在写任何代码或敲定任何设计之前，**必须先看至少一份真实的业务凭据（订单/合同/纸单等）**。我们花了 4 轮提问才发现的认知偏差，看一眼订单文件 5 分钟就能定位。

### 0.5 系统元定位（⚡ 2026-05-22 元决策）

**业务方原话：** "系统主要目的是统计展现订单生产的状态，方便追踪生产情况，也是为了领导层能够看到和管控生产流程中间的成本和浪费"

**这是一个元决策——决定本系统不是什么的同时，决定它是什么：**

| 系统是 | 系统不是 |
|---|---|
| ✅ 事实记录器（每日实绩、件数偏差、成本数据） | ❌ 排程引擎（不优化生产顺序） |
| ✅ 异常放大器（损耗高、延期、工价偏差自动浮出） | ❌ 决策替代器（人定大方向，系统不抢） |
| ✅ 成本镜子（让领导看见浪费在哪） | ❌ APS / MES 系统（不做精细派工） |
| ✅ 滚动预测器（预计出单日动态刷新） | ❌ 承诺生成器（不锁日期，按实际节奏走） |

**派生原则：**

1. **双日期模型——合同日是承诺，系统日是预测**

   | 日期 | 性质 | 来源 | 是否变化 |
   |---|---|---|---|
   | **合同交货日** | 死线、承诺 | 合同 / 啤机部规定 | ❌ 不变 |
   | **预计出单日** | 预测、镜子 | 系统算 = `(剩余目标数 ÷ 报价目标数) + 今天` | ✅ 每天刷新 |

   - 实际产量每天波动 → 预计出单日每天刷新
   - 系统**持续对比"预计 vs 合同"**：预计 > 合同 → 飘红预警"以当前节奏将晚 X 天交付"
   - 业内类比："物流的预计送达日" / "rolling ETA"

2. **排期由人，系统呈现**
   - 车间主管定订单顺序 → 系统不自动排
   - 拉长定每天喷什么颜色 → 系统不进车间内细节
   - 系统的甘特图是"主管已经决定的事的可视化"，不是"系统推荐你这么做"

3. **不建产能模型**
   - 不需要"车间每天能做 X 件"的硬性产能上限
   - 不需要"产能冲突检测"
   - 预估产能直接复用工艺模板的"报价目标数"（详见 1.6）

4. **成本数据必须可视化**
   - 工价偏差（实绩工价 vs 模板核定）
   - 损耗率（按工序、款号、操作员等下钻）
   - 半成品库存价值（散件 + 超额库存）
   - 外发延期影响（约定返厂 vs 实际返厂）

**这条元决策对后续所有章节的设计约束：**
- 第 2 章排期：粗粒度，不做产能/产线优化
- 第 5 章上机：只记开工/完工事实，不下"开工指令"
- 第 6 章实绩：以"记录"为本，不以"控制"为本
- 第 9 章日报：以"对比核定值"为本（成本镜子）

### 0.6 跨章节核心概念：统一库存机制（2026-05-22 浮出，第 3 天扩展为半成品+成品统一）

**问题来源：** 第 2 章 Q10 + Q7 损耗监督讨论时，业务方提到三个场景：
1. **木桶效应散件**——100 套兔子做完后，耳朵 100 / 身体 90 / 腿 95，多出 10 个耳朵 + 5 个腿
2. **炒货机超额生产**——超小件用炒货机喷色，一锅几十万件，本订单只用一部分
3. **成品入库**——每日良品数自动累加入成品库存，等装配部取货

三个场景**本质相同**：都是 (款号, 版本, 子件, 部位, 颜色) 维度的进出账。**合并为一张统一 `inventory` 表 + 状态字段。**

**核心机制：**

```
inventory（统一库存表）
  主键：(款号, 版本, 子件, 部位, 颜色)
  数量字段：可用数量
  状态字段：
    - 半成品（未走完所有工序）
    - 成品（印喷所有工序完成，等装配部取货）
    - 散件（成品但不成套，等翻单复用）
  来源字段：每日实绩入账 / 木桶散件 / 炒货机超额 / 其他
  时间字段：入库日 / 上次出库日
```

**进账（库存增加）：**

| 来源 | 触发时机 | 数量 |
|---|---|---|
| **每日良品入账** | 每天 daily_outputs 录入后 | 良品数自动累加 |
| **木桶效应散件**（被动）| 订单完工时 | 各部位完工数 > 木桶最小值的差额 |
| **炒货机超额**（主动）| 开工决定"凑满一锅" | 超出订单需求的部分 |

**注：不良数不入库**（业务方决策）—— 只在 daily_outputs 上记录，供 QC 部追踪不良率。

**出账（库存减少）：**
- **翻单消耗**：新订单录入时，系统按 (款号, 版本, 子件, 部位, 颜色) 查询库存，提示"库存有 X 件可用"
  - 散件复用范围：**仅限同 (款号, 版本) 内**（Q13 决策）
- **装配部取货**：成品被装配部领走 → 减库存
- **报废处理**：长期不用的散件由人工标记报废

**散件 vs 成品的区分逻辑（V1 简化版）：**
- 系统每日完工时自动判断：
  - 该 (部位, 颜色) 的累计良品数 ≤ 套件 MIN → 进"成品"状态
  - 超出套件 MIN 的多余部分 → 进"散件"状态
- 详细计算放第 4 章实施

### 1.1 订单录入方式（✅ 已确认）

### 1.3 订单数据结构（✅ 已确认 + 第 2 天追加部位层和版本字段）

### 1.4 「子件」与「部位」(✅ 第 2 天追加部位层)

**3 层模型（强制始终 3 层）：**

```
款号 + 版本
  └ 子件 (该款号下的独立生产单元)
      └ 部位 (该子件的物理部位)
```

#### 1.4.1 两种业务场景（占比约 5:5）

| 场景 | 子件举例 | 部位举例 |
|---|---|---|
| **套装产品** | 兔子 / 青蛙 / 蝾螈（套装小件） | 兔子耳朵 / 兔子身体 / 兔子腿 |
| **单玩具** | 手 / 脚 / 头（本身就是部位） | 手→手 / 脚→脚 / 头→头（自指部位） |

**关键决策（A1）：单玩具场景也强制建"同名自指部位"。**
- ✅ 数据模型统一（永远 3 层）
- ✅ 同一款号里可以**混合两种场景**（业务方确认会有），不需要分支处理
- ✅ UI 上看似冗余但能"看到真实情况"（业务方原话）

**单玩具场景的录入辅助：** 文员录入时部位字段**默认带出子件名**，一键确认即可，不增加操作负担。

#### 1.4.2 「同名子件不跨款号复用」(Q4 sub-a 决策)

业务事实：即使两个款号都叫"兔子"，造型可能完全不同（站姿 vs 蹲姿），工艺也不同。
- 工艺模板按 `(款号, 版本, 子件, 部位)` 联合键
- 主数据维护时**不建"全局子件库"**，避免误用
- 翻单时只能在同一 `(款号, 版本)` 内复用，跨款号不行

#### 1.4.3 木桶效应（✅ 第 3 天扩展为两层）

**两层 MIN：**

1. **部位内（多工序）**：部位完工数 = `MIN(各工序步骤完工数)`
   - 例：兔子头 移印 100、喷油 80、洗油 60、补油 50 → 部位完工 = 50
   - 因为多工序顺序执行，瓶颈工序决定部位真正完工数

2. **部位间（套件）**：整套件可成品数 = `MIN(各部位完工数)`（A4 决策）
   - 例：兔子耳朵 50、身体 45、腿 60 → 整套兔子可成品 = 45

**为什么两层都要：**
- 部位内多工序 MIN — 用于 daily_outputs → inventory 入账时判断"散件 vs 成品"
- 部位间套件 MIN — 用于订单完工率 / 预计出单日预测 / 木桶散件入库判断

#### 1.7.5 实绩金额计算公式（v0.6 业务方拍板最终版）

**业务方原话**："实绩金额公式用总核价算，算实际成本给领导层看的，报价是参考"

```
某条 daily_outputs 的金额（实际成本视角）
  = 良品数 × (该步骤总核价 + (该步骤 color_upcharges 中匹配本批颜色名的加价 || 0))
  = 良品数 × (labor_price + production_cost + paint_cost + 该步骤命中色加价)

其中：
  - 步骤总核价 = labor_price + production_cost + paint_cost
  - 颜色匹配：daily_outputs.color_name == steps[i].color_upcharges[j].color_name
  - 不匹配则加价 = 0（普通色无加价）
```

**为什么用总核价不用报价：**
- 总核价 = 部门**实际花掉的钱**（人工 + 生产 + 油漆）→ 给领导看的"实际成本"
- 报价 = 给客户的**对外价**，是事先谈定的固定数字
- 累计实际成本 vs 累计报价 = 部门盈亏（业务方原话"如果成本长期增加会影响是否和客户加价"）

- ✅ 人工价是步骤级，已经分布在每条 daily_outputs 上——**不需要额外归属规则**

## 第 2 章：订单排期（✅ 第 3 天基本完成）

> **范围：** 订单从"已录入"到"已上车间"之间的排期决策
> **元定位提醒：** 排期由人决定，系统呈现。所有日期都是滚动预测，不是承诺（详见 0.5）

### 2.1 谁来排期？（✅ 已确认）

| 角色 | 职责 | 范围 |
|---|---|---|
| **车间主管 × 2**（兴信A 一位 + 华登A 一位）| 决定订单进哪天上车间、订单顺序 | 订单级粗粒度 |
| **拉长 × N**（每车间多位组长）| 决定每天喷什么颜色、谁喷、版材切换 | **车间内细节，不进系统** |

**不存在"中央计划员"角色**——主管两人各自管自己车间的甘特图。系统只协助车间主管，不替主管做决定。

**业务方原话：** "排期具体的单就是由车间主管排，在细节的地方就是由拉长自己掌控"

### 2.2 车间分配规则（✅ 已确认）

| 规则 | 内容 | 来源 |
|---|---|---|
| **UV 硬约束** | 工艺模板含 `UV` 类型工序步骤 → **必须分到兴信A** | Q5 + 业务方原话"兴信A 有 UV，华登A 没有" |
| **非 UV 订单** | 兴信A / 华登A 均可，由文员/主管手动选 | 自由分配 |
| **款号惯性** | 系统**可**显示该款号历史上去过哪个车间作建议（V1 可选，V2 必做）| 经验沉淀 |

**自动识别 UV 的依据：** 工艺模板的"工序步骤类型标签"（详见 1.6）——如果模板中存在 `类型=UV` 的步骤，系统自动建议/强制兴信A。

### 2.3 甘特图占位模型（✅ 已确认）

**主视图：粗粒度订单级**

```
                  5-22  5-23  5-24  5-25  5-26  5-27  ...
兴信A 车间    ━━━[ZWZ2026057 17子件]━━━  [ZWZ2026059]━━
华登A 车间    [ZWZ2026058]━━━  ━━[ZWZ2026060 9子件]━━━━
                                          🔴 插单
外发订单池    [W2026012 万达] 约定返厂 5-26  已交付 60%
```

**展示规则：**

| 维度 | 决策 |
|---|---|
| 行（Y 轴）| 车间（兴信A / 华登A / 外发订单池）|
| 列（X 轴）| 时间（按天）|
| Bar 起点 | **上车间日期（人工指定）**|
| Bar 终点 | **预计出单日（系统动态算）**= (剩余目标数 ÷ 每天预估产能) + 今天 |
| Bar 颜色 | 订单状态（待开工 / 生产中 / 已完工 / 延期 / 急单插入）|
| Bar 长度变化 | 实际产量高于预估 → bar 自动缩短；低于预估 → bar 自动延长 |

**150+ 行订单怎么展示？** 主视图每订单 1 条 bar。**点击 bar → 弹出订单详情面板**展开 6 层结构（子件/部位/颜色/工序/进度）。详情面板沿用第 1 章数据结构。

**外发订单池的展示：**
- 独立分组（不占车间产能行）
- 显示约定返厂日 / 实际累计交付百分比 / 是否影响主订单出单日

### 2.4 优先级与插单（✅ 已确认）

**默认排序：** 按交期升序（最近交期排前）。

**插单流程：**
1. 车间主管在甘特图上**拖入新订单**（或从订单列表"插入"）
2. 系统**只记录新顺序**，不自动顺延后续订单的日期
3. 系统**自动重算所有订单的预计出单日**（基于每天的实际产能数据滚动刷新）
4. 受影响订单的 bar 自动延长 → 主管可见"延期红色预警"

**关键原则：不存在"插单顺延算法"**——因为系统从未承诺过具体日期。

**业务方原话：** "其实每天的产量不一定是固定的，有时候可能产量高一点，原本 10 天的单 8 天完成了，这种变化由人工掌控，系统只按照每天预估产量计算出单日期，日期可以根据每天的实际生产数变化"

系统的角色是"持续反映当下事实"，不是"承诺一个日期再去守"。

**急单标记：** 车间主管可手动标记某订单为"急单"（视觉上飘红），用于区分。

### 2.5 外发订单的"排期"（✅ 已确认 + 修正 0.2 节）

### 2.7 订单汇总表（✅ 第 3 天新增，拉长录入入口）

> **范围：** 区别于甘特图，订单汇总表是**生产数录入的入口**，由拉长每日使用。

#### 2.7.1 两张表的分工

| 视图 | 谁看 | 用途 |
|---|---|---|
| **甘特图**（详 2.3）| 车间主管 | 排单视角——订单跨度、状态、车间分布、预计/合同日期对比 |
| **订单汇总表** | 拉长 | 录入视角——全订单列表 + 分拉聚合 + 每日生产数录入 |

#### 2.7.2 录入流程（拉长视角）

```
拉长 → 进入"订单汇总表"
     → 选自己的拉（拉 = 拉长 + 工艺步骤 + 机器类型）
     → 系统列出该拉今天要做的所有 (订单, 部位, 颜色)
     → 对每条逐项录入：机台号 / 计划数 / 实际数 / 不良数 / 工时 / 人数 / 备注
     → 保存 → 写入 daily_outputs（详见 1.9 表清单）
```

**核心特性：**
- **录入最小粒度** = `(拉, 机台号, 订单, 部位, 颜色, 日期)` 一行
- **工艺步骤**自动推断（拉绑定步骤标签，拉长不需要选）
- **客名 / 款号 / 部位 / 总订单数 / 余下数**由系统从订单带出，拉长不需要重复填
- **机台号**必填（V1 与纸单粒度一致，便于拉长追踪自己车间内的机台进度）
- **工资聚合**天然按拉划分——每条 daily_outputs × 该拉工艺步骤单价 = 该拉的产值

#### 2.7.3 多道工序自然产生多条记录

举例：兔子头 工艺模板有 `移印 ×2 → 喷油 ×1 → 洗油 ×1 → 补油 ×1` 五道工序。

| 日期 | 拉 | 机台号 | 部位 | 实际数 | 工艺步骤（自动）|
|---|---|---|---|---|---|
| 5-22 | 胡旗移印拉 | 6# | 兔子头 | 200 | 移印（第1遍）|
| 5-22 | 胡旗移印拉 | 6# | 兔子头 | 180 | 移印（第2遍）|
| 5-23 | 王某喷油拉 | 12# | 兔子头 | 150 | 喷油 |
| 5-24 | 李某洗油拉 | 33# | 兔子头 | 100 | 洗油 |
| 5-25 | 赵某补油拉 | 48# | 兔子头 | 80 | 补油 |

部位完工数 = MIN(各工序完工数) = MIN(200, 180, 150, 100, 80) = 80（详见 1.4.3 木桶两层）

#### 2.7.4 监督 4 维（沿用 0.5 元定位的"成本镜子"）

| 监督维度 | 系统怎么做 | 主要受众 |
|---|---|---|
| **计划 vs 实际** | 每日计划数 vs 实际数，偏差大或连续未达 → 预警 | 车间主管 / 拉长 |
| **预计 vs 合同** | 预计出单日 vs 合同交货日，超过 → 飘红 | 主管 / 领导 |
| **良品 vs 不良** | 不良率按 部位 / 机台 / 拉长 / 客名 聚合排行 | QC 部 / 领导 |
| **外发约定 vs 实际** | 约定返厂 vs 累计交付，逾期/滞后预警 | 主管 / 领导 |

#### 2.7.5 与旧日报系统的关系

业务方现有系统的日报（含 报价/油请价/工价/产值/无产值/浪费工资 等字段）—— **本系统不复制它，只提供数据源**：
- 本系统出原始数据：`daily_outputs` 明细 + 库存进出账
- 旧系统拿这些自己算工资/产值/损耗等衍生字段
- 业务方原话："做好系统后，日报数据自然就出来了"
- 对接方式（API / 导出文件 / 数据库直连）属第 11 章实施时再定
